# Embue — Secure IoT Data Architecture & Decentralized Integrity Layer

## Overview

At Embue, I worked as one of the Blockchain Leads on a team proof of concept for securing and verifying IoT telemetry generated by environmental sensors deployed in apartment buildings.

Embue’s sensors collect environmental data such as temperature, humidity, and related signals used for energy analytics, operational monitoring, and building efficiency workflows. The core systems problem was that centralized IoT storage creates single points of failure, weak auditability, and limited guarantees that stored sensor data has not been altered.

I helped design and prototype a decentralized data infrastructure layer using **GnuPG encryption, Web3.Storage, IPFS, and Filecoin**. The system encrypted IoT sensor data before storage, uploaded encrypted files through Web3.Storage, used IPFS content addressing for retrieval, and leveraged Filecoin-backed storage concepts to improve auditability and tamper resistance.

## Technical Problem

Embue’s IoT network relied on centralized infrastructure for storing and managing device data. This created several technical risks:

* A single point of failure for sensor data
* Weak guarantees that stored records had not been altered
* Limited auditability around when data was stored, transferred, or accessed
* Security risk if one compromised device or system exposed broader network data
* Difficulty sharing selected data with external stakeholders while preserving confidentiality

The core challenge was not simply storing IoT data. The challenge was designing a system where telemetry could be encrypted, stored, verified, retrieved, and selectively shared without relying entirely on a centralized data store.

Because blockchain systems are public and poorly suited for storing large volumes of raw telemetry directly on-chain, the architecture needed to separate several concerns:

* **Confidentiality:** Sensor data needed to be encrypted before storage
* **Storage:** Large telemetry files needed off-chain storage
* **Integrity:** Stored data needed tamper-evident references
* **Verification:** The system needed stronger guarantees around persistence and auditability
* **Access Control:** External data requests needed a structured approval workflow
* **Usability:** The system needed to integrate with Embue’s existing IoT workflow without requiring a full infrastructure overhaul

## Architecture Design

The proof of concept used a layered architecture to separate responsibilities across the system.

The architecture was organized into four primary layers:

* **Data Input Layer**
* **Storage Layer**
* **Extension / Access Control Layer**
* **Application Layer**

This separation was important because each layer solved a different systems problem.

The **data input layer** defined what data entered the system and how that data was prepared before storage. For Embue, this meant IoT telemetry from environmental sensors.

The **storage layer** handled encrypted file persistence and decentralized retrieval. Instead of storing raw data directly on-chain, the architecture used Web3.Storage, IPFS, and Filecoin-related infrastructure.

The **extension layer** modeled how access-control logic could coordinate stakeholder requests, admin approval, and retrieval behavior.

The **application layer** represented the interaction surface between Embue, external stakeholders, and the underlying decentralized infrastructure.

The main architectural decision was to avoid treating blockchain as the storage layer for all raw data. Direct blockchain storage was not practical for large IoT data volumes. Instead, the architecture used a hybrid design:

* **GnuPG** protected confidentiality before storage
* **Web3.Storage** managed upload and storage coordination
* **IPFS** provided content-addressed retrieval using CIDs
* **Filecoin** provided decentralized persistence and verification concepts
* **Smart-contract workflow design** modeled request, approval, and denial logic
* **Application-layer flow** connected the technical system to real stakeholder interaction

This allowed the proof of concept to use decentralized infrastructure for auditability and integrity while keeping bulk data storage off-chain.

## Data Input & Encryption Layer

The data input layer prepared Embue’s IoT telemetry for secure decentralized storage.

Because decentralized storage systems such as IPFS are publicly retrievable by content identifier, raw sensor data could not be uploaded directly. The data first needed to be encrypted so that only authorized parties could interpret the contents after retrieval.

The team used **GnuPG** to encrypt files before storage. GnuPG supports public-key cryptography, where:

* A public key encrypts the data
* A private key decrypts the data
* Encrypted data can be stored on public infrastructure without exposing the underlying contents
* Different stakeholders can receive access to different encrypted datasets based on key management

This encryption layer was critical because Filecoin and IPFS do not automatically provide privacy. They provide decentralized storage and content addressing, but confidentiality must be handled before upload.

In the proof of concept, a small portion of Embue sensor data was encrypted before being uploaded through Web3.Storage and later retrieved through the IPFS/Filecoin-backed workflow.

## Decentralized Storage Layer

The storage layer handled encrypted file upload, content addressing, and decentralized retrieval.

The team selected **Filecoin** for the proof of concept because of its decentralized nature, immutability, low energy consumption, and relatively low storage fees. However, the architecture did not treat Filecoin as a place to directly store all raw IoT telemetry at scale. Instead, the system used Web3.Storage to coordinate the storage workflow across decentralized infrastructure.

The storage process worked as follows:

1. Embue sensor data was collected for the proof of concept
2. The data was encrypted using GnuPG
3. The encrypted file was uploaded through Web3.Storage
4. Web3.Storage converted the file into a content identifier, or CID
5. The CID made the file addressable through IPFS
6. Filecoin-backed storage infrastructure supported persistence and verification
7. The encrypted file could later be retrieved using the CID and decrypted with the appropriate private key

This design provided tamper evidence through content addressing. If the contents of a stored file changed, its CID would change as well. That made the file reference itself a form of integrity check.

The architecture therefore separated:

* **Privacy:** GnuPG encryption
* **Upload coordination:** Web3.Storage
* **Content addressing:** IPFS CIDs
* **Decentralized persistence:** Filecoin-backed storage
* **Access:** Private-key-based decryption and modeled stakeholder workflows

## Filecoin, Web3.Storage & IPFS Integration

A major part of the project was understanding how Filecoin, Web3.Storage, and IPFS interact.

Web3.Storage acted as the practical interface for uploading encrypted files. It abstracted away much of the complexity of dealing directly with IPFS and Filecoin infrastructure.

IPFS provided content-addressed retrieval. Instead of retrieving data from a location-based URL, the system retrieved data using a CID generated from the file contents. This means the identifier is tied to the data itself, not just to where the data is hosted.

Filecoin provided the persistence and verification layer. The purpose of Filecoin in the architecture was not to replace every storage system with direct blockchain storage, but to strengthen auditability and persistence for data stored through decentralized infrastructure.

This distinction was important. The architecture used:

* **IPFS** for content-addressed distribution and retrieval
* **Filecoin** for decentralized persistence and proof-oriented storage concepts
* **Web3.Storage** as the API layer connecting the application workflow to IPFS/Filecoin infrastructure

This reduced integration complexity and made the proof of concept more practical for a startup environment.

## Access Control & Smart Contract Workflow Design

The proof of concept also explored how external stakeholders could request access to selected IoT data.

The architecture included a modeled smart-contract-based access workflow. This was not positioned as a fully deployed production smart contract system. It was a design for how access decisions could be structured and automated in a future implementation.

The modeled access workflow worked as follows:

1. An external stakeholder requests access to specific IoT data
2. Embue reviews the request as the system administrator
3. If Embue approves the request, the access workflow initiates the retrieval path
4. The encrypted data is retrieved using the relevant CID
5. The authorized stakeholder decrypts the data using the appropriate private key
6. If Embue denies the request, the retrieval path is terminated and the user is notified

This design helped translate blockchain infrastructure into a real business process. Instead of treating blockchain as an isolated technical layer, the architecture connected decentralized storage to stakeholder access, approval logic, and data-sharing requirements.

## Application Layer

The application layer represented the user-facing workflow connecting Embue, external stakeholders, and the decentralized storage system.

This layer mattered because the system needed to be more than a technical storage demo. It needed to show how Embue could use decentralized infrastructure in a real business context involving secure data sharing, auditability, and selective access.

## End-to-End Proof of Concept

The team successfully produced a working proof of concept using a small portion of Embue sensor data.

The validation workflow involved:

* Gathering sample Embue sensor data
* Encrypting the data using GnuPG
* Uploading encrypted files through Web3.Storage
* Generating IPFS-compatible CIDs
* Storing data through Filecoin-backed decentralized infrastructure
* Retrieving files from IPFS using the generated CID
* Decrypting retrieved files using private-key-based access
* Documenting the system flow through architecture diagrams

This proof of concept demonstrated three core properties:

* **Confidentiality:** Data was encrypted before entering public decentralized infrastructure
* **Integrity:** CID-based retrieval made stored content tamper-evident
* **Auditability:** Filecoin-backed storage concepts provided a stronger basis for persistence and verification than centralized storage alone

The result was not a production deployment. It was a validated blueprint that Embue and Cyvl could build on for future secure data-sharing and auditability use cases.

## Alternative Platform Evaluation

The project evaluated several blockchain and decentralized storage platforms before settling on the proof-of-concept architecture.

Platforms explored included:

* **Filecoin**
* **Web3.Storage**
* **IPFS**
* **Storj**
* **Skynet**
* **Polkadot**
* **Substrate**

Substrate and Polkadot were explored as potential blockchain development paths. A custom proof-of-existence architecture using Substrate could theoretically store file hashes, ownership records, and block numbers on-chain. However, this would introduce significant complexity because it would require custom blockchain infrastructure rather than leveraging existing decentralized storage networks.

The final architecture favored Filecoin, Web3.Storage, and IPFS because they provided a more practical path for the project constraints:

* Faster proof-of-concept implementation
* Lower infrastructure burden
* Better alignment with decentralized storage use cases
* Stronger fit for content-addressed data retrieval
* Less need to build and maintain a custom blockchain system

This evaluation reinforced the hybrid design: use decentralized storage and verification infrastructure where it adds value, but avoid unnecessary blockchain complexity.

## Engineering Challenges

The project involved several real-world engineering constraints.

### Time Constraints

The blockchain proof of concept was developed within a seven-week MQP project. This required narrowing the scope to a minimum viable prototype focused on secure storage, retrieval, and auditability.

### Privacy Constraints

IPFS content is publicly retrievable by CID. This made encryption before upload mandatory. GnuPG became a critical part of the architecture because decentralized storage alone did not solve confidentiality.

### Scalability Constraints

Directly storing large volumes of IoT telemetry on-chain was not practical. The architecture needed to separate bulk storage from verification and auditability.

### Integration Complexity

The project required understanding how GnuPG, Web3.Storage, IPFS, Filecoin, smart-contract workflow design, and application-layer access requests fit together. Misunderstanding one component would distort the overall architecture.

### Environment Setup

The blockchain development environment involved tools such as WSL, Ubuntu, Node.js, NPM, and decentralized storage APIs. Setting up and documenting the environment was part of making the proof of concept reproducible.

### Platform Selection

The team evaluated multiple decentralized storage and blockchain platforms before selecting the most practical architecture for the project timeline and use case.
